# 15. 작업 세션 다루기

> **쉽게 말하면:** 큰 일일수록 계획을 먼저 세우고, AI와의 대화(맥락)는 얇고 한 목표로 유지한다.

AI(Claude)와 오래 작업할 때 결과 품질과 속도를 지키는 규칙.

## 계획 먼저

- 3파일 이상 / 구조 변경 / 비가역 작업 / 모호한 요청은 **코드보다 계획이 먼저**다.
  `plan-feature` 스킬로 인터뷰 → 단계 계획 → 확인 후 착수.
- 작은 단일 변경은 계획을 건너뛰고 바로 해당 스킬로 간다. 과하게 격식 차리지 않는다.

## 맥락은 얇게, 한 세션 한 목표

- 한 대화 세션은 **하나의 목표**에 집중한다. 목표가 바뀌면 대화를 새로 시작(`/clear`)한다.
- 대화가 길어져 느려지거나 앞 내용을 놓치기 시작하면 요약해서 이어 간다(`/compact`).
- 지금 작업과 무관한 파일을 잔뜩 열어 두지 않는다 — 맥락이 흐려지면 엉뚱한 곳을 고친다.

## 자율성과 가드의 균형

- 명백한 기본값이 있는 선택은 되묻지 말고 진행하되 **무엇을 했는지 밝힌다**(`rules/05`).
- 단, **비가역·승인 필요 작업에서는 멈춘다**: 보호 브랜치 머지(`rules/70`), 임의 패키지·구조 변경
  (`rules/10`), 시크릿·환경(`rules/80`). 가드가 명확하면 그 안에서는 자율적으로 진행해도 된다.

## 단계마다 검증·정직하게

- 큰 작업은 작은 단계로 쪼개 **각 단계 후 검증**하고(`rules/50`), 다음으로 넘어간다.
- "다 됐다"고 단정하지 말고 검증된 영역 / 미검증 영역을 나눠 보고한다(`rules/05`).
- 가설이 틀렸으면 부분 땜질보다 되돌리고 다시 본다(`debug-issue`).

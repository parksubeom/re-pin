# 변경 리뷰

커밋/PR 전에 변경을 규칙 기준으로 점검한다. `docs/ai/pr-review-checklist.md`의 리뷰어 항목을
실제로 실행하는 스킬이다.

## 1. 범위 파악
```
git diff            # 작업 트리
git diff --staged   # 스테이징
git diff main...HEAD  # 브랜치 전체
```
변경 파일과 계층(app/pages/…/shared)을 먼저 파악한다.

## 2. 점검 차원 (규칙 매핑)
| 차원 | 무엇을 보나 | 규칙 |
| --- | --- | --- |
| 범위 | 요청 범위 밖 리팩터링/구조 변경이 섞였나 | 10 |
| 아키텍처 | 계층 import 방향, public API(index.ts) 경유, 깊은 경로 직접 접근 | 20 |
| 컴포넌트 | `interface Props`/cva/cn/Tailwind 전용, `<style>` 없음 | 30 |
| 상태/API | 서버 상태가 Pinia로 샜나, 생성 타입 사용, 쿼리 키 colocation | 40 |
| 테스트 | 신규 컴포넌트에 spec+stories, 로직/가드 변경에 회귀 스펙 | 50 |
| 에러/UX | 토스트가 두 레이어에서 중복 발생하나, 메시지 SSoT | 60 |
| 보안 | 시크릿 하드코딩, `.env` 변경, 위험 DOM 주입 | 80 |
| 접근성 | 시맨틱 요소·라벨·키보드·포커스 | 35 |
| 정확성 | 반대 방향 실패 모드(안티 가설), 엣지 케이스 | — |
| 정리 | 중복·재사용 여지·불필요한 복잡도 | — |

## 3. 지적 방식
- 각 지적은 **`file:line` + 심각도(High/Medium/Low) + 한 줄 문제 + 고칠 방향**.
- 확신이 없는 지적은 "확인 필요"로 표시하고 단정하지 않는다(안티 가설로 스스로 반증 시도).
- 파괴적 부수효과(자동 삭제·저장·정리 가드)를 바꾼 변경은 **반대 방향 회귀**를 특히 본다.
- 억제(eslint-disable, ts-ignore)로 통과시킨 흔적이 있으면 지적.

## 4. 마무리
- High가 하나라도 있으면 PR/커밋 전에 해결 권고.
- 리뷰 결과를 검증된 것/추정인 것으로 분리해 보고(`rules/05`). 필요하면 `write-test`로 회귀 스펙 제안.

# 기능 계획 (인터뷰 → 계획 → 확인 → 위임)

코드부터 짜지 않는다. **의도를 먼저 명확히** 하고, 작은 단계로 쪼갠 계획을 사람이 확인한 뒤
실행한다. `rules/05-working-with-claude.md`(범위 확인)와 `rules/15-working-sessions.md`(계획 먼저)를 실행한다.

## 언제

- 요청이 모호하거나, 3파일 이상/구조 변경/비가역 작업이거나, "무엇을 만들지"가 기술 용어로
  정리되지 않은 경우(비개발자 요청 포함).
- 작은 단일 변경이면 이 스킬을 건너뛰고 바로 해당 build 스킬로 가도 된다.

## 단계

### 1. 인터뷰 — 의도 캐기 (일상어)

정답을 넘겨짚지 말고 짧게 묻는다:

- **누가 / 무엇을 / 왜**: 누가 쓰나, 어떤 정보를 보고 무엇을 하나, 왜 필요한가.
- **입력·출력**: 무엇을 넣으면 무엇이 나와야 하나.
- **엣지/예외**: 데이터가 없을 때·에러일 때·권한 없을 때는?
- **완료 기준**: "이러면 done"이라고 부를 수 있는 조건.

### 2. 계획으로 번역 (기술)

의도를 우리 구조에 매핑한다:

- 어떤 **슬라이스/컴포넌트/API/테스트**가 필요한가(`rules/20·30·40·50`).
- **작은 수직 슬라이스**로 쪼갠다 — 가장 얇게 "끝에서 끝까지 도는 한 조각(tracer bullet)"을 1단계로,
  이후 단계로 살을 붙인다. 한 번에 다 벌이지 않는다.
- 각 단계에 어떤 스킬을 쓸지 표기: `gen-api-types` → `create-slice` → `create-component` → `write-test`.

### 3. 자기비평 (critic)

계획을 스스로 반증한다(`review-changes` 정신):

- 빠진 화면/상태/에러 케이스는? 범위 밖으로 새는 변경은?(`rules/10`)
- 파괴적 부수효과나 반대 방향 실패 모드는? 더 단순한 방법은?

### 4. 확인 (일상어 미리보기)

계획을 **전문용어 없이** 요약해 사용자에게 보여주고 승인받는다:

- "① 사용자 목록 API 연결 → ② 목록 화면 → ③ 검색창 → ④ 테스트" 식.
- 비가역/큰 변경은 반드시 명시적 승인 후 진행(`rules/10·70`).

### 5. 위임 & 검증

승인된 계획을 단계별로 해당 build 스킬에 넘긴다. **각 단계 후 검증**(`rules/50` runbook),
문제가 생기면 다음 단계로 넘어가기 전에 해결. 진행 상황은 검증/미검증으로 나눠 보고(`rules/05`).

## 산출물

코드가 아니라 **합의된 계획**(단계 목록 + 각 단계 스킬 + 검증 방법). 이후 실행은 build 스킬이 맡는다.

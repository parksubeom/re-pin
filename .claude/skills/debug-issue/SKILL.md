# 버그 디버깅

추측으로 코드부터 고치지 않는다. **재현하고 가설을 검증한 뒤** 최소한으로 고친다.

## 1. 재현

- 정확한 **입력·환경·기대 결과 vs 실제 결과**를 확보한다.
- 재현이 안 되면 먼저 재현 경로부터 찾는다(어떤 조건에서만 나는지).

## 2. 가설

- 코드 경로를 추적해 원인 후보를 좁힌다.
- 힌트: **같은 입력인데 경로별로 결과가 다르면** 자원(개수·크기)보다 **코드 경로 분기**를 먼저 의심.
  (예: "만들 때는 되는데 불러올 때는 안 됨" → 두 경로의 초기화 차이)

## 3. 검증 (고치기 전에)

- 최소 실험·로그·임시 테스트로 가설을 **먼저 확인**한다. 성능 민감 구간은 조건부 플래그로 로그를 감싼다.
- 가설이 코드 경로와 안 맞으면 수정하지 말고 가설을 다시 세운다.

## 4. 최소 수정

- 확인된 원인만 최소 범위로 고친다. 요청 밖 리팩터링·구조 변경 금지(`rules/10`).
- "관찰 지점에서 막기"(싸고 국소적) vs "근본 원인 제거"(비싸고 광범위)를 비용/위험으로 판단해 택한다.

## 5. 회귀 스펙 + 정리

- 같은 버그가 다시 나지 않게 **회귀 테스트**를 추가한다(`write-test`). 파괴적 가드 변경이면 반대
  방향 케이스도.
- 수정이 가설에 기반했는데 **가설이 틀린 것으로 드러나면**, 부분 복원 대신 되돌리고(그 가설로 만든
  스펙도 함께 제거) 깨끗한 상태에서 재조사한다.

## 검증·보고

`rules/50` runbook으로 확인하고, 검증된 것/미검증인 것을 나눠 보고한다(`rules/05`).
